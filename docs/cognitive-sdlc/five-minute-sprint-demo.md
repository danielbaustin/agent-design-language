# Five-Minute Sprint Demo

## Status

Tracked demo plan for the C-SDLC throughput proof surface.

## Purpose

The five-minute sprint is the flagship C-SDLC demo. It should demonstrate that
governed parallel engineering cognition can reduce coordination latency without
weakening review, replay, merge, closeout, or memory truth.

## Demo Thesis

A bounded software change can be decomposed into parallel cognitive shards,
executed under explicit governance, converged into evidence, reviewed, and
prepared for merge faster than a traditional sequential issue flow.

The claim is not that AI writes code quickly. The claim is that C-SDLC reduces
the serial coordination fraction while preserving governance.

## Required Proof

A credible demo must show:

- transition identity
- issue-local `SIP -> STP -> SPP -> SRP -> SOR`
- shard plan or transition DAG
- explicit writable-path ownership
- evidence bundle
- review synthesis
- merge-readiness gate
- tracked workflow records
- trace or signed-trace-ready proof references
- closeout and ObsMem handoff boundary
- measured timing

## Demo Boundaries

The demo must not:

- bypass human review authority
- bypass CI or branch protection
- merge unreviewed work silently
- hide failed validation
- use speed to excuse stale cards or missing closeout
- claim full organizational automation

## Repeatability Bar

One successful demo is not enough for default operation.

Before C-SDLC is claimed as operationally reliable, the five-minute sprint lane
should run more than once, record metrics for each run, and preserve residual
risks when the process needs repair.
