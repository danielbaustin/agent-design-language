# Next Milestone Handoff - v0.91.2

## Purpose

Record what `v0.91.2` hands forward after the tooling, evaluation,
productization, publication, and repo-health band closes.

This is the `WP-23` next-milestone planning surface. It preserves the standard
release-cycle shape: the issue immediately before release ceremony is planning
for what comes next, not another implementation issue.

## Immediate Handoff: v0.91.3

The next milestone is `v0.91.3`.

`v0.91.3` is the first C-SDLC implementation milestone. It should consume the
corrected card lifecycle, workflow guardrails, review-skill lessons, and release
tail evidence from `v0.91.2` to prove one bounded Cognitive State Transition.

`v0.91.4` remains the downstream C-SDLC hardening and default-operation
milestone. It should consume the `v0.91.3` first slice and make the process
repeatable enough to be the normal ADL software-development path.

This insertion does not cancel the already planned `v0.92` identity/birthday
work or `v0.93` constitutional citizenship/social-cognition work. It means
those milestones should start from a stronger development control plane.

## v0.91.3 Inputs From v0.91.2

`v0.91.3` should consume:

- the corrected `SIP -> STP -> SPP -> SRP -> SOR` lifecycle contract
- workflow-conductor, editor-skill, worktree, review, and closeout discipline
- the docs-only and code-issue lessons from the v0.91.2 sprint-conductor runs
- accepted review-remediation evidence from the internal and third-party review
  cycle
- healthier focused-validation and test-cycle expectations
- review packet, evidence packet, and product-report surfaces from CodeFriend
  work
- workflow guardrails for main-branch writes, hung closeout watchers, and unsafe
  report generation
- tracked ADR planning for C-SDLC tracked workflow state and signed trace
  boundaries

## Required v0.91.3 Direction

`v0.91.3` should stay vertical. Its job is to prove one governance-complete
Cognitive State Transition, not to turn the entire project into a new platform
in one pass.

The expected first-slice surfaces are:

- Cognitive State Transition manifest
- transition DAG and shard/barrier model
- issue-local operative `SPP` execution plan truth
- evidence bundle and review synthesis shape
- governed merge-readiness gate that preserves GitHub issue, PR, CI, and human
  review truth
- SRP review-result and SOR outcome-truth handoff to ObsMem
- first five-minute-sprint proof surface and metrics snapshot

## v0.91.4 Completion Context

`v0.91.4` is not the next milestone, but the `v0.91.3` plan should preserve a
clean handoff into it. The `v0.91.4` completion milestone should make the first
slice dependable:

- stricter lifecycle validators
- aligned `pr doctor`, workflow-conductor, sprint-conductor, and editor routing
- shard ownership and interface-freeze enforcement
- repeatable evidence convergence and review synthesis
- merge-readiness and PR-gate hardening
- tracked durable workflow records under the documented workflow namespace
- minimal signed trace proof for durable C-SDLC runs
- ObsMem transition-memory integration
- repeatability metrics for more than one five-minute sprint
- active-issue migration policy and process-drift regression fixtures

## Downstream Consumers

- `v0.92` identity, continuity, birthday, ACP, and ACIP binary/WebSocket work
  should benefit from the C-SDLC control plane rather than receiving raw
  v0.91.2 workflow debt.
- `v0.93` citizenship, Theory of Mind, social cognition, delegation, IAM, and
  enterprise-security work should inherit better evidence, trace, review, and
  closeout mechanics.
- Later product, review, publication, and repo-visibility work should reuse the
  review/evidence surfaces instead of reconstructing them from local lore.

## Non-Reduction Rule

- `v0.91.2` should not replace `v0.91.1`
- `v0.91.2` should not silently absorb future identity or governance milestones
- `v0.91.3` and `v0.91.4` should not absorb GWS, GTM, or unrelated product work
  into C-SDLC
- C-SDLC should remain public, tracked, auditable workflow truth; local `.adl/`
  state may support execution but is not sufficient as durable proof

## Release-Ceremony Boundary

This handoff does not claim `v0.91.2` release approval. `WP-24` still owns the
release ceremony, release evidence, and end-of-milestone report.
