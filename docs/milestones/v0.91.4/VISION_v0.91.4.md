# v0.91.4 Vision

## Metadata

- Project: `Agent Design Language`
- Milestone: `v0.91.4`
- Version: `v0.91.4`
- Date: `2026-05-22`
- Owner: ADL maintainers
- Related issues: `#3100`, planned issue wave pending

## Purpose

Define the milestone-level vision for completing the C-SDLC rollout after the
`v0.91.3` first slice.

## Overview

`v0.91.4` is the milestone where ADL moves from one proved Cognitive State
Transition to repeatable default operation.

The release should establish:

- strict lifecycle validation and routing
- reliable conductor/editor/doctor agreement
- Software Development Polis actor standing sufficient for repeated execution
- shard ownership and interface-freeze enforcement
- repeatable evidence convergence, review synthesis, and merge-readiness gates
- tracked durable workflow records
- minimal signed trace proof
- ObsMem ingestion from tracked evidence
- five-minute-sprint repeatability metrics

The goal is not more theory. The goal is a dependable process that future ADL
software-development issues can actually use.

## Core Goals

`v0.91.4` advances ADL in five areas:

1. Lifecycle and routing hardening.
2. Transition operation.
3. Sprint default behavior and repeatability metrics.
4. Durable tracked workflow state, signed trace proof, and memory integration.
5. Review, remediation, planning, and release discipline.

## Lifecycle And Routing Hardening

The milestone should make process drift harder to reintroduce:

- validators reject or route stale lifecycle states
- conductor and doctor agree about workflow state
- editor skills repair cards without opportunistic hand edits
- active issue migration policy is explicit
- process-drift regression fixtures cover known failure modes

## Transition Operation

The C-SDLC must become operational:

- participants have explicit actor roles and standing boundaries
- shard ownership and interface freeze rules are enforceable
- evidence bundles converge across transition steps
- review findings and residual risks remain visible
- merge-readiness gates preserve GitHub truth

## Durable State, Trace, And Memory

By the end of the milestone, durable C-SDLC truth should be tracked in Git under
the documented workflow namespace. Local `.adl` state may support execution, but
it cannot be the only proof surface.

Signed trace proof and ObsMem ingestion should make transition history
inspectable, replay-aware, and memory-ready.

## Five-Minute Sprint Repeatability

`v0.91.4` should show the five-minute sprint is not a one-off miracle. It should
measure repeated bounded transitions and record coordination-latency evidence
without weakening review or governance.

## Milestone Context

`v0.91.3` proves the first slice. `v0.91.4` turns the slice into the default
issue execution lane. Later milestones can build product and social-cognition
features on that stabilized base.

## Long-Term Direction

C-SDLC is intended to remodel software development around tracked intention,
bounded execution, reviewable evidence, and truthful memory. `v0.91.4` is the
moment where that model must become ordinary enough to rely on.
