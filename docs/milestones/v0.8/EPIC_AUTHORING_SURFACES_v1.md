# EPIC: Authoring Surfaces v1  
**Version:** v0.8  
**Area:** authoring  
**Status:** Draft  
**Issue:** #517  

---

# 1. Executive Summary

ADL currently requires users to author schema-compliant YAML manually.  
This creates a high barrier to entry and limits adoption to users already familiar with the ADL spec and deterministic runtime model.

Authoring Surfaces v1 introduces a user-facing authoring layer that enables:

- Plain-English → ADL workflow generation
- CLI guided workflow scaffolding
- Constrained LLM-assisted generation
- Deterministic canonical YAML emission
- Safe defaults enforcement

This EPIC does **not** change ADL’s runtime determinism, signing, trust, or artifact guarantees.

Cards remain internal contributor workflow artifacts and are not part of the end-user product surface.

---

# 2. Motivation

ADL’s strength is deterministic orchestration and auditable artifacts.

However:

- Users must currently write YAML directly.
- Users must understand retry semantics, concurrency rules, and failure taxonomy.
- There is no ergonomic front door to ADL.

To become broadly usable, ADL must support non-coding authoring workflows while preserving its contract-driven identity.

The goal of this EPIC is to create a clean front door to ADL.

---

# 3. Product Positioning

After this EPIC, ADL should function as:

> A deterministic workflow contract layer that can be generated from natural language.

Comparable analogy:

- Kubernetes YAML is rarely handwritten.
- OpenAPI specs are often generated.
- Terraform plans are scaffolded.

ADL YAML should be:

- Canonical
- Deterministic
- Reviewable
- Generated by tooling

---

# 4. Architectural Model

Authoring Surfaces v1 introduces a new layered structure:

## Layer 1 — User Authoring Interface (New)

- CLI guided workflow creation
- Natural-language prompt → workflow generation
- Optional LLM assistance
- Schema-aware scaffolding

## Layer 2 — Canonical Authoring AST (New)

- Structured intermediate JSON model
- Deterministic ID assignment
- Pattern registry integration
- Safe default injection
- Strict validation before YAML emission

## Layer 3 — ADL YAML (Existing)

- Canonical spec format
- Deterministic formatting
- No random IDs
- Stable ordering

## Layer 4 — Runtime (Existing)

- Deterministic ExecutionPlan
- Stable artifact model
- Signing / trust enforcement
- Resume + resilience surfaces

LLM output is never written directly to YAML.  
LLM output must first pass through the Authoring AST layer.

---

# 5. Design Principles

1. Determinism is non-negotiable.
2. Authoring must not weaken trust or policy.
3. YAML emission must be canonical and stable.
4. LLMs may suggest — they may not define runtime semantics.
5. Users must not need to understand cards, artifacts, or contributor workflow.

---

# 6. Non-Goals (v1)

- Full visual DAG editor
- Distributed cluster execution
- Memory subsystem expansion
- Tool marketplace
- Provider refactor
- Schema changes

This EPIC strictly adds a front-end authoring surface.

---

# 7. Initial Work Packages

## WP-01 — Canonical Authoring IR (Workflow Cards → IR → YAML)

Define a user-facing **Authoring IR** (intermediate representation) that is *not* the runtime schema.

This IR is the canonical target for:
- UI form/wizard authoring
- “Structured prompt” (card) authoring
- Constrained LLM output

Key idea:
- **LLMs and UIs never emit runtime YAML directly.**
- They emit the Authoring IR (JSON), which ADL validates and deterministically compiles into canonical ADL YAML.

Minimum IR fields (v1):
- workflow metadata (name, description)
- steps (id, title, prompt, inputs/outputs)
- dependencies / DAG edges
- retry policy (attempts, classification)
- on_error policy (fail/continue)
- concurrency hints (bounded parallelism)
- approval gates (human confirmation) as an explicit step kind
- timeouts (optional)

Determinism requirements:
- Deterministic ID assignment (stable ordering, no randomness)
- Canonical ordering of steps/edges
- Stable formatting for emitted YAML

Deliverables:
- Authoring IR schema (JSON Schema)
- Deterministic compiler: Authoring IR → canonical ADL YAML
- Golden tests for stable emission

---

## WP-02 — CLI Guided Generator (No LLM)

Introduce a guided authoring CLI that produces **Workflow Cards / Authoring IR** and then emits canonical ADL YAML.

Commands (proposed):
- `adl init workflow` (wizard)
- `adl emit` (IR/cards → YAML)

Wizard capabilities (v1):
- Choose a workflow pattern (linear, fork/join)
- Add steps (title + prompt)
- Configure retry/on_error defaults
- Add approval gates
- Emit bundle

Outputs:
- Authoring IR JSON (source of truth)
- Canonical ADL YAML (generated)

No LLM required in this phase.

---

## WP-03 — Constrained LLM Generator (Experimental)

Introduce natural-language-to-workflow generation behind a constrained interface.

Command (proposed):
- `adl generate "<plain english>"`

Pipeline:
1. LLM returns **Authoring IR JSON** only (never YAML)
2. Validate JSON against Authoring IR schema
3. Deterministically canonicalize (IDs, ordering, defaults)
4. Emit canonical ADL YAML

Failure behavior:
- Invalid JSON or schema mismatch produces a deterministic error with actionable remediation.

---

## WP-04 — Pattern Registry Integration

Expand and formalize a small set of deterministic workflow patterns that the authoring surfaces can compose.

Patterns (starter set):
- linear
- fork_join
- conditional
- retry
- approval_gate
- timeout

Deliverables:
- Pattern definitions (authoring-level)
- Deterministic expansion rules
- Tests that verify stable expansions

---

## WP-05 — Safe Defaults Framework

Define a safe baseline configuration that authoring tools apply when the user leaves options unspecified.

Defaults must:
- Avoid unbounded concurrency
- Avoid infinite retries
- Avoid silent continuation on critical failures
- Avoid policy/trust weakening

Deliverables:
- Default profiles (e.g., "safe", "dev")
- Deterministic default injection rules
- Documentation describing defaults

---

## WP-06 — PWA Structured Prompt Editor (Scaffold)

Create a minimal **PWA** that acts as a “structured prompt editor” for ADL.

Positioning:
- Cards are “structured prompts” (fields + constraints), not freeform YAML.
- The editor generates Authoring IR and emits canonical ADL YAML.

PWA v1 capabilities:
- Form + wizard UI to author steps, dependencies, retries, approval gates
- Import/export Authoring IR JSON
- Export canonical ADL YAML

Explicit constraints (v1):
- No drag/drop DAG editor required
- No remote execution required
- No accounts/multi-user required

Deliverables:
- Static PWA (JS/TS) with local-only storage
- IR schema-driven form rendering (simple)
- Export bundle (IR + YAML)

---

## WP-07 — Local Run Bridge (Optional; sets up v0.9 Rust rewrite)

Provide a small local “bridge” process so the PWA can run and monitor ADL workflows without requiring the browser to execute arbitrary commands.

This is a stepping stone toward a Rust rewrite (target v0.9).

Capabilities (v1):
- Local-only HTTP or IPC endpoint
- Whitelisted commands:
  - `adl validate <file>`
  - `adl run <file>`
- Stream run status from `.adl/runs/<run_id>/run_status.json` (or equivalent)

Security posture:
- Localhost only
- Explicit user consent (startup)
- No secret handling in the bridge

Deliverables:
- Minimal bridge prototype (language TBD)
- Spec for a future Rust implementation

# 8. Risks and Mitigations

## Risk: Authoring surfaces weaken determinism
Mitigation: all generation goes through Authoring IR → deterministic canonical emitter. No direct YAML emission from LLM/UI.

## Risk: Unsafe workflows generated
Mitigation: Safe Defaults Framework + schema validation + deterministic error messages.

## Risk: Confusion between contributor cards and user-facing cards
Mitigation: clearly distinguish:
- **PR cards**: internal contributor workflow artifacts under `.adl/cards/<issue>/`
- **Workflow cards**: user-facing structured prompts that compile into workflows

## Risk: PWA cannot run local commands
Mitigation: PWA v1 is generate/export only; optional Local Run Bridge provides a safe, constrained execution channel.

---

# 9. Success Criteria

- A user can author a workflow without writing YAML.
- The authoring surface emits canonical ADL YAML that passes `adl validate`.
- The workflow structure is deterministic and reviewable.
- LLM assistance (if used) is constrained to Authoring IR output.

---

# 10. Long-Term Vision (Beyond v1)

- Visual DAG editor (drag/drop)
- VSCode extension
- Rust/WASM authoring engine shared with the runtime (single source of truth for canonical emission)
- Import/export adapters from other agent frameworks
